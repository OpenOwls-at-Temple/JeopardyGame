# Owl Jeopardy — LLM Integration Design

> **Owner: Wei**
> This document describes the design, rationale, and extension points for the
> AI question-generation pipeline. It covers what is built in Phase 2, what
> trade-offs were made, and how Phase 3 (slide ingestion) builds on top of it.

---

## 1. Overview

The LLM integration is a **question producer**. Its job is:

```
(topic, categories, point values) → Question[]
```

That's the entire contract with the rest of the app. The game board, scoring
engine, and bank editor do not know or care whether a question was typed by a
teacher or generated by Claude.

---

## 2. What is built (Phase 2)

### 2a. Core module: `src/lib/aiGenerate.ts`

```
Entry point:  generateQuestions(opts: GenerateOptions): Promise<Question[]>

Input:
  opts.topic        string   — subject matter ("World War II", "Photosynthesis")
  opts.categories   string[] — column headers the teacher wants on the board
  opts.pointValues  number[] — one question per (category × point value) pair
  opts.apiKey       string   — Anthropic API key (client-side for now)

Output:
  Question[]  — shaped exactly like the domain type; ready to pass to AppContext
```

### 2b. UI: `src/components/AIGenerateModal.tsx`

A two-screen modal on the Bank Editor page:

**Screen 1 — Input**
- Anthropic API key (password-masked; optional save to localStorage)
- Topic / subject
- Categories (comma-separated; pre-filled from the bank's existing categories)
- Point values (comma-separated; defaults to 100, 200, 300, 400, 500)
- Generate button

**Screen 2 — Review**
- Table of all generated questions with checkboxes
- Teacher can deselect any questions they don't want
- "Add N Questions" button writes selected questions into the bank

**Screen transitions:**
```
[Input] --Generate clicked--> [Loading spinner] --success--> [Review]
                                                 --error--> [Input + error msg]
[Review] --Back button--> [Input]
[Review] --Add N Questions--> [Bank editor, modal closes]
```

## 3. Prompt design

This is the most important part of the LLM integration. The quality of generated
questions is entirely determined by the prompts.

### System prompt (static, set once)
```
You are a Jeopardy question writer for classroom review games.
When given a topic and categories, produce valid Jeopardy-style clues where the
"question" field is the clue shown to players and the "answer" field is the
expected response. Keep clues clear and educationally appropriate.
Always respond with valid JSON only — no markdown fences, no extra text.
```

**Why this system prompt works:**
- Sets the persona ("Jeopardy question writer") so the model uses the right register
- Clarifies the field semantics ("question" = clue, "answer" = response) — without
  this the model might swap them or use question-mark phrasing everywhere
- Hard constraint on output format (JSON only) reduces parse failures
- "Educationally appropriate" nudges away from trick questions or ambiguous clues

### User prompt (dynamic, built per generation)
```
Generate Jeopardy questions for a classroom review game.
Topic: "{topic}"
Categories: {categories as JSON array}
Point values to use (one question per value per category): {pointValues as JSON array}

Return a JSON array of objects with this shape:
{ "category": string, "question": string, "answer": string, "points": number }

Rules:
- Create one question per (category × point value) combination.
- Higher point values should be harder questions.
- The "question" field is the clue players see (not phrased as a question).
- The "answer" field is the expected answer.
- Stay on the topic: "{topic}".
```

**Why the explicit rules matter:**
- "One question per (category × point value)" gives the model a precise count to
  target — without this you get an unpredictable number of questions
- "Higher point values should be harder" maps the Jeopardy convention onto the model
- Repeating the topic at the end reduces category drift on long generations

### Response parsing
1. Try `JSON.parse(text)` directly
2. If that throws, strip markdown fences (` ```json ... ``` `) and retry
3. If still invalid JSON → throw `Error` with message shown to user in the modal
4. Validate that the result is an array before proceeding

## 4. Model selection

| Model | Cost | Speed | Quality | Use case |
|-------|------|-------|---------|----------|
| claude-haiku-4-5 | ~$0.00025/1K input | Fast | Good | **Current default** |
| claude-sonnet-4-6 | ~$0.003/1K input | Medium | Excellent | Power users / complex topics |
| claude-opus-4-8 | ~$0.015/1K input | Slow | Best | Not needed here |

**Current choice: Haiku.** A typical generation (3 categories × 5 point values = 15
questions) fits comfortably in one request at roughly 400 input tokens + 800 output
tokens ≈ $0.001 per generation. Fast enough that the teacher does not feel they are
waiting.

**When to upgrade to Sonnet:** if teachers report that generated questions are too
simple, contain errors, or do not match the topic well. Add a `model` option to
`GenerateOptions` and let the UI expose a "Quality" toggle.

## 5. API key handling

### Current (Phase 2): client-side
- Teacher enters their own Anthropic API key in the modal
- Optional save to `localStorage` under key `owl_jeopardy.anthropic_key`
- The key is sent directly in the `Authorization` header from the browser
- Requires the `anthropic-dangerous-direct-browser-calls: true` header

**Risk:** anyone who opens DevTools can read the key from the request headers or
localStorage. Acceptable for a classroom prototype where the teacher controls the
device, but not for a multi-user deployment.

### Future (Phase 3 or production): server-side proxy
Replace the `fetch` in `aiGenerate.ts`:

```
Current:  fetch('https://api.anthropic.com/v1/messages', { headers: { 'x-api-key': apiKey } })
Future:   fetch('/api/ai/generate', { body: JSON.stringify({ topic, categories, pointValues }) })
```

The proxy endpoint (Express / Next.js API route / Cloudflare Worker) holds the key
in an environment variable and forwards the request to Anthropic. The UI never sees
the key.

**`GenerateOptions` change:** drop `apiKey` from the interface. No other changes.

## 6. Error states and UX handling

| Error | Cause | User-facing message |
|-------|-------|---------------------|
| Empty topic | User did not fill the field | "Please enter a topic." |
| Empty categories | All commas, no text | "Please enter at least one category." |
| Empty API key | Field blank | "Please enter your Anthropic API key." |
| 401 Unauthorized | Wrong API key | "Claude API error: invalid x-api-key" |
| 429 Too Many Requests | Rate limit hit | "Claude API error: rate_limit_exceeded" |
| JSON parse failure | Model returned non-JSON | "Unexpected response shape from Claude" |
| Network error | No internet | Browser-native fetch error message |

All errors surface in the modal above the action buttons. The teacher can correct
the input and retry without closing the modal.

## 7. Phase 3 extension: slide ingestion

Phase 3 adds a new input path: teacher uploads a PPTX or PDF and the LLM generates
questions from its content. The `generateQuestions` function stays identical — only
the topic/context changes.

### Planned module: `src/lib/slideParser.ts`

```typescript
// Extracts readable text from an uploaded file.
// Returns an array of text chunks (one per slide or page).
parseSlide(file: File): Promise<string[]>
```

**PPTX:** use a library like `pptx-parser` or `jszip` + manual XML extraction to
pull text nodes from slide XML.

**PDF:** use `pdf.js` (Mozilla's browser-native library) to extract text layer.

### Modified generation prompt for Phase 3

When slide text is available, inject it as context:

```
[existing user prompt]

Use the following lecture content as your source material:
---
{slideChunks.join('\n\n')}
---
Generate questions that test understanding of this specific content.
```

**Chunking strategy:** if the extracted text exceeds ~4000 tokens, split into chunks
and run multiple generation calls (one per chunk), then merge and de-duplicate the
`Question[]` arrays before showing the review screen.

### De-duplication
After merging chunks, remove near-duplicates:
- Exact match on `question` string → drop one
- Same `answer` + same `category` → flag for review (show with a ⚠️ badge)

## 8. Future ideas (not planned, for discussion)

### Streaming generation
The Claude API supports streaming (`stream: true`). Instead of showing a spinner,
the review table could fill in row by row as questions arrive. Good UX for large
generation requests.

Implementation: use the `anthropic` npm SDK's streaming helper rather than raw
`fetch`, and update the `preview` state array incrementally.

### Teacher feedback loop
Add a thumbs-up/thumbs-down button next to each generated question in the review
table. Store feedback in localStorage. Future: batch-send feedback to a backend to
fine-tune prompt quality over time, or simply use it to filter the current batch.

### Difficulty calibration
Add a `difficulty` param (`easy | medium | hard`) to `GenerateOptions`. Inject into
the system prompt: "For easy questions, ask for definitions. For hard questions,
require application of concepts."

### Caching
Cache the last N generation results in localStorage (keyed by `{topic}:{categories}:{pointValues}`).
If the teacher generates questions for the same topic twice, show the cached result
instantly and offer a "Regenerate" option. Reduces API cost during iteration.
